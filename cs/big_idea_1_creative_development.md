# Big Idea 1: Creative Development (CRD)

**Exam Weight:** 10-13%

## Overview

Programming is a creative activity that drives innovation across many fields. Building something meaningful typically involves investigating needs, designing solutions, prototyping ideas, and testing the result. Collaboration and continuous iteration are the core mechanisms that push a project from rough draft to finished product.

## Key Topics

### Collaboration Improves Computing Outcomes

Different people bring different perspectives and skills, which makes software more complete and inclusive.

- A computing innovation always has software at the heart of what it does
- Computing innovations take many forms: physical devices (like self-driving cars), pure software applications (like photo editors), or abstract computing concepts (like e-commerce)
- When developers come from diverse backgrounds, their varied viewpoints tend to produce more well-rounded and accessible results
- Common collaboration methods include pair programming, team-based development, and peer review
- Staying in contact with end users throughout the design process helps ensure the innovation actually addresses real needs

### Tools and Benefits of Working Together

Online tools let developers collaborate effectively across distance:

- Version control systems, shared document platforms, and real-time communication tools dramatically improve cross-team efficiency
- Working with others surfaces perspectives that any single developer might miss, often leading to better solutions
- Collaboration also reduces the risk of bias that can creep in when only one viewpoint shapes the design

### Interpersonal Skills in Collaboration

Effective collaboration goes beyond merging code — it requires:

- Clear communication so everyone understands goals and progress
- Building consensus when opinions diverge, and resolving conflicts constructively
- Negotiation and compromise so different ideas can converge into a unified direction
- Collaboration skills affect not only how the work gets done but also how understandable the final product is to others

### Iterative Design Process

Good programs aren't written in one pass — they're refined through repeated cycles of building, testing, and revising.

#### Clarifying the Purpose of an Innovation

Before writing any code, ask: **what problem does this solve?**

- The purpose of a computing innovation is to solve a problem or fulfill an interest through creative expression
- A clear understanding of purpose helps developers make design decisions that actually fit the situation

#### Basic Building Blocks of a Program

- A **program** is a collection of instructions that runs on a computer to accomplish a specific task (also called software)
- A **code segment** is any subset of instructions within a larger program
- A program's **behavior** describes how it operates during execution — typically expressed in terms of how users interact with it

#### Inputs and Outputs

- **Program input**: data sent to a program for processing; can come from user actions or other programs
- **Event**: an occurrence that triggers corresponding input data to be sent into the program
- **Program output**: any data the program sends out — as visuals, sound, haptic feedback, or text

#### Designing the Development Process

A well-structured process makes programs more reliable:

- Development processes can be methodical or exploratory, depending on the project
- Common phases include: investigating requirements, designing the architecture, building a prototype, and testing
- **Iteration** means continually refining based on test results and feedback, rather than demanding perfection from the start
- **Incremental development** lets developers implement and test one small piece at a time, reducing the risk of late-stage failures

#### User Interface Design

- Designing a program starts with understanding what users actually need and expect
- A **specification** defines all the requirements a program must satisfy
- User interfaces should prioritize intuitive usability above everything else

#### Program Documentation

- Documentation helps developers maintain working code and lets others understand it quickly
- Documentation can include: descriptions of what the code does, contributor lists, external references, and change history
- **Comments** are documentation written in the programming language itself, embedded directly in the code

#### Crediting External Code

- Using code, algorithms, or ideas written by others should always include a clear attribution of the source
- Building on existing libraries or open-source code is standard industry practice

## Key Terms

| Term | Definition |
|------|------------|
| Computing innovation | A creative output — device, software, or concept — that has a program as its core functional component |
| Iterative development | A development approach that continuously refines a solution through repeated rounds of testing and feedback |
| Incremental development | A strategy of breaking a program into small pieces, implementing and testing each piece before moving on |
| Pair programming | Two programmers working together at one computer — one writes, the other reviews in real time |
| Program documentation | Written explanations that describe the purpose and logic of code |
| User interface (UI) | The layer through which a user interacts with a program |
| Specification | A detailed document that defines all the requirements a program must meet |
| Prototype | An early working version of a program used to test and validate design direction |

## Common Misconceptions

1. **Programming is solitary work** — Modern software development relies heavily on collaboration. Even "solo" projects typically use libraries and tools built by others.
2. **Once a program is written, it's done** — Software development is an ongoing iterative process; programs need continuous revision, testing, and improvement.
3. **Only the final product matters** — The process itself — planning, testing, documentation — is just as important as the end result. A solid process is what produces a reliable product.
4. **Using someone else's code is always wrong** — Borrowing libraries or open-source code is the industry standard. What matters is properly crediting the source and respecting the license terms.
5. **Documentation is optional** — Good documentation is essential for maintenance and team collaboration. Undocumented code is often only understandable by the person who wrote it.
